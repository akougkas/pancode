# Wave 3: Worker Coordination Surface

## Goal

Register session commands (/context, /reset, /checkpoint) so operators can inspect
and manage coordination state. Wire the dispatch loop to sync the board after each
worker returns. After this wave, the coordination layer is observable and controllable.

## Depends On

- Wave 1A (session substrate: context-registry, shared-board, memory)
- Wave 1B (worker launch: safety-ext with coordination tools)
- Wave 1-fix (runtime root consistency)

## Files to Modify

### 1. `src/domains/session/extension.ts`

Add command registration. Session domain owns /context, /reset, /checkpoint.

**Add commands:**

`/context [key]` (read-only view of context registry):
- No args: list all context entries with source, timestamp, and truncated value (80 chars)
- With key arg: show full value for that key
- Display format: table with columns Key, Source, Timestamp, Value
- Import renderers from ui domain for consistent formatting

`/reset [context|all]` (clear coordination state):
- No args: clear shared board (all namespaces) + temporal memory. Print confirmation.
- `reset context`: clear context registry. Require confirmation via `ctx.ui.confirm()`.
- `reset all`: clear board + temporal + context. Require confirmation.
- Print count of cleared entries for each store.

`/checkpoint [list|restore <id>]` (lightweight session markers):
- No args: save checkpoint. Use `pi.appendEntry("pancode:checkpoint", { ... })` with:
  - timestamp, budget snapshot, context entry count, board entry count, run stats
- `checkpoint list`: list all saved checkpoints from session entries
- `checkpoint restore <id>`: rehydrate PanCode state from checkpoint marker
  (log the checkpoint data, but do not auto-restore state in Phase B; display only)

**Implementation pattern for commands:**
```typescript
pi.registerCommand("context", {
  description: "View the PanCode context registry",
  handler: async (args, ctx) => {
    const registry = getContextRegistry();
    if (!registry) {
      ctx.ui.notify("Context registry not initialized.");
      return;
    }
    // ... render context entries
  },
});
```

**Update session_start to also sync board before logging:**
After creating the three stores, call `sharedBoard.sync()` to pick up any
entries left by workers from a previous session.

### 2. `src/domains/dispatch/extension.ts`

After each worker returns (in both dispatch_agent and batch_dispatch), sync the
board to pick up worker writes:

```typescript
// After worker completes, before updating run status:
const { getSharedBoard } = await import("../session");
const board = getSharedBoard();
if (board) board.sync();
```

This ensures orchestrator-side board state reflects worker writes.
Use dynamic import to avoid adding session as a hard dependency of dispatch
(dispatch depends on safety and agents, not session).

**Also: pass agentName through to spawnWorker.**
The dispatch_agent tool already has the agent name. Pass it as `agentName` in
SpawnOptions so the worker knows its identity for board/context writes.

### 3. `src/domains/ui/extension.ts`

Update the categorized /help rendering. Group commands by domain:

This replaces the flat list. Each category is a header line followed by indented

Add a footer line: `Pi native: /new /compact /fork /tree /session /export /resume`

## What NOT to Do

- Do NOT register tools in the session extension. Worker tools are already in safety-ext.ts.
- Do NOT add cross-domain state mutation. Session commands read/clear session-owned state only.
- Do NOT change the manifest dependencies. Session stays at dependsOn: [].
- Do NOT implement full checkpoint restore logic. Phase B checkpoint is display-only.
