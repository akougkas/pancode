# v0.1.5: Shadow Agents

## Goal

Add shadow_explore as an orchestrator-internal tool for pre-dispatch codebase
intelligence. Shadow agents are tools available to the orchestrator to collect
information quickly before making dispatch decisions. They run in-process,
use cheap/fast models, and are never exposed to users or workers.

## Locked Decision

Shadow agents are orchestrator-internal reasoning tools. They are NOT part of
the dispatch system. They help the orchestrator explore the codebase, understand
the request, and make better dispatch plans. Scout is the first shadow agent.
Dev and reviewer remain subprocess workers.

Implementation: Pi tool with in-process agent. Register shadow_explore as a
Pi tool via registerTool(). When the orchestrator calls it, PanCode runs a
fast LLM call using a cheap/local model with exploration-specific instructions
and read/grep/find tools. Returns structured findings.

## What to Build

### 1. `src/domains/agents/shadow-explore.ts` (NEW)

Register a `shadow_explore` tool that the orchestrator LLM calls naturally:

```typescript
pi.registerTool({
  name: "shadow_explore",
  description: "Explore the codebase to gather intelligence before dispatching
    work. Use this to understand file structure, find relevant code, and orient
    yourself before planning dispatch.",
  parameters: Type.Object({
    query: Type.String({ description: "What to explore" }),
  }),
  async execute(id, params, signal, onUpdate, ctx) {
    // Run a lightweight Pi agent session in-process
    // Uses the cheapest available local model
    // Has read/grep/find/ls tools only (readonly)
    // Returns structured exploration findings
  },
});
```

### 2. Shadow Agent Model Selection

The shadow agent should use the cheapest/fastest available model:
- Prefer local small models (qwen3:0.6b, gemma3:1b)
- Fall back to the orchestrator model if no small model available
- Configurable via PANCODE_SHADOW_MODEL env var

### 3. Orchestrator Prompt Wiring

Update the orchestrator system prompt to tell it about shadow_explore:

```
For any internal tasks that will increase your understanding of the request
and the codebase, use shadow_explore to investigate before dispatching work
to agents. Shadow exploration is fast and uses a lightweight model. Use it
to: count files, find patterns, read key configs, understand project structure.
```

## Reference

- Mastracode subagents: explore/plan/execute (in-process, not subprocess)
- Pi SDK createAgentSession: can create a nested session for the shadow
- Current agents: `src/domains/agents/` (agent spec registry, teams)
- Old scout concept: lightweight readonly exploration agent

## What NOT to Do

- Do NOT expose shadow_explore to workers
- Do NOT add shadow agents to the dispatch system
- Do NOT use the orchestrator's expensive model for shadow work
- Do NOT make shadow agents visible in /agents list (they are internal tools)
